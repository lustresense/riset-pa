# Poin-Poin Keputusan Bu Hesti (Bimbingan 16/6/26)

Sumber utama: Transkrip bimbingan 16 Juni 2026 (terbaru & paling penting)
Sumber pendukung: Bimbingan 13/3/26 & 20/4/26 (hanya jika masih relevan)

---

## 1. User Flow 3 Fase — Sudah Benar Di-Split
User flow dibagi menjadi 3 fase sudah disetujui Bu Hesti. Pembagian fase sudah tepat, tinggal perbaikan shape dan format.
- **Sumber:** 16/6/26 — "Ya kamu udah split ya? Iya saya udah split. Oke good."

## 2. Shape Flowchart Harus Benar (Wajib Patuh Standar)
Shape di flowchart tidak boleh salah: **Input = jajaran genjang (parallelogram), Proses = persegi panjang (rectangle), Decision = belah ketupat (diamond), Start/End = oval.** Ini standar basic ALPRO yang wajib dipatuhi.
- **Sumber:** 16/6/26 — "Kamu nggak boleh keliru menggunakan logic shape-nya. Ini logic shape-nya di programming kan harus seperti ini."

## 3. Hapus Kata "From" di Diagram
Jangan tulis "from fase 1" atau "from fase X" di flowchart. Langsung tulis nama fasenya saja.
- **Sumber:** 16/6/26 — "Gak usah pakai kata from. Oh gitu. Jadi langsung fase satu gitu?"

## 4. Font Harus Diperbesar
Font di semua diagram yang akan di-submit harus diperbesar agar bisa dibaca.
- **Sumber:** 16/6/26 — "Font-nya digedein. Nanti kalau kamu submit ini font-nya kurang, kurang besar nanti dibesarkan font-nya."

## 5. Warna Harus Sama/Konsisten Antar Fase
Warna di flowchart harus konsisten per fase (misal fase 1 = oren, dst). Jangan campur aduk.
- **Sumber:** 16/6/26 — "Warnanya betul. Harus sama warnanya. Ini dari fase satu tadi yang oren."

## 6. Desain Sistem SATU Global — Jangan Di-Split Dua
Desain sistem dibuat satu diagram besar keseluruhan, bukan di-split menjadi diagram terpisah antara Can dan Dias. Pekerjaan masing-masing dibedakan dengan **warna**, bukan diagram terpisah.
- **Sumber:** 16/6/26 — "Jangan di-split jadi dua. Satu sistem besar, bedakan pekerjaan kamu dan Diaz pakai warna. Main warna yang paling enak."

## 7. Use Case = Aktor (Stickman) + Aktivitas — BUKAN Flow
Use case yang kemarin **salah total** karena dibuat seperti flow. Use case yang benar: ada aktor (stickman wajib format standar ALPRO) di sisi kiri, aktivitas di sisi kanan. Satu kotak = satu aktivitas, TIDAK menunjukkan urutan/flow. Hanya menunjukkan siapa bisa akses aktivitas apa.
- **Sumber:** 16/6/26 — "Enggak. Kamu kok nggak paham-paham sih. Use case itu harus ada orang dan harus ada aktivitas. Intinya itu. Stickman harus seperti ini. Formatnya harus. Fix ini. Bukan saya ngarang."

## 8. Gameplay Progress Feedback = Decision, Bukan Proses Lurus
Di fase 3, gameplay progress feedback harus menggunakan shape decision (belah ketupat). Kalau yes → lanjut level berikutnya, kalau no → balik ke fase 2 (redraw).
- **Sumber:** 16/6/26 — "Gameplay progress feedback. Apakah ada? Berarti opsi dong. Iya. Opsi di sini decision dong."

## 9. Login WAJIB — User Harus Bisa Save History
User **wajib login**, bukan hanya session ID. Karena total waktu bermain ~30 menit terlalu lama untuk sekali jalan tanpa save. User butuh history (pernah main atau belum, progress sebelumnya).
- **Sumber:** 16/6/26 — "Mending pakai login. Login dah. Iya. Harus ada history kalau memang skenarionya seperti itu. Iya. Harus ada history."

## 10. Maskot = Teks Bubble Saja, Tanpa Suara/NLP
Penjelasan edukasi di akhir level menggunakan **bubble teks** dari maskot. Tidak pakai suara, tidak pakai NLP. Maskot cukup jadi auto-text biasa.
- **Sumber:** 16/6/26 — "Teks aja sih, Bu. Jadi kita nggak pakai suara gitu. Oh oke. Iya kayak bubble gitulah." | Dukungan 20/4/26 — "Kalau nggak perlu ya, kalau dia hanya sekedar text aja kan ya tidak ada yang lain."

## 11. Hapus Dekoratif — Pakai Solid & Danger Saja
Kategori objek **"dekoratif/tembus pandang"** dihapus dulu karena fungsinya masih ragu. Yang fix dipakai hanya **solid** dan **danger**. Dekoratif bisa ditambah nanti sambil jalan kalau sudah jelas konsepnya.
- **Sumber:** 16/6/26 — "Kalau masih ragu mendingan jangan. Karena ini sambil jalan bisa bertambah. Kalau kamu rasa oh bisa nih akhirnya menemukan konsep untuk dekoratif fungsinya buat apa. Tuh nah itu ditambah kan."

## 12. Objek Berbahaya Gunakan "Duri", Bukan Pisau
Untuk objek danger, gunakan visual **duri (thorns)** alih-alih pisau atau senjata tajam. Lebih soft dan aman untuk konteks anak SMP.
- **Sumber:** 16/6/26 — "Duri ya ya udah lebih soft ya. Iya. Oh oke ya udah nggak apa-apa."

## 13. Level Design: 3 Level Saja, Cukup — Jangan Tambah
Tiga level sudah banyak. Tidak perlu menambah. Level 1 target 2 menit (gampang), level makin tinggi makin menantang (bukan makin cepat).
- **Sumber:** 16/6/26 — "Tiga udah banyak loh. Saya biasanya dua aja udah oke. Tapi kalau digeser sampai tiga mungkin singkat-singkat ya."

## 14. Tutorial: General, Jangan Guide Step-by-Step
Informasi/tutorial di level awal harus **general** saja, jangan mengguide step-by-step. Kalau terlalu detail, gamenya jadi tidak berguna karena user sudah tahu caranya. Level 1 boleh ada clue, level 2-3 kurangi clue.
- **Sumber:** 16/6/26 — "Jangan sampai itu mengguide mereka untuk mengerjakan game karena jadinya nanti nggak ada gunanya kamu bikin game kalau mereka sudah tahu step by step. General ya."

## 15. Metodologi WAJIB Ada — Sebelum Implementasi
Metodologi penelitian harus muncul sekarang, bukan belakangan. Tanpa metodologi, proposal tidak terstruktur. Strukturnya: studi literatur → analisis kebutuhan → implementasi (per fase).
- **Sumber:** 16/6/26 — "Metodologi itu harus muncul. Harusnya metodologi dulu. Iya biar metodologi itu akan lebih terstruktur."

## 16. Fishbone untuk Analisis Tahapan
Buat diagram **fishbone** untuk menunjukkan urut-urutan tahapan pengerjaan (studi literatur, desain maskot, mockup, dll). Ini wajib untuk proposal.
- **Sumber:** 16/6/26 — "Bikin fishbone-nya. Supaya bisa kelihatan urut-urutan tahapannya itu bagaimana."

## 17. Desain Sistem = Format INPUT–PROSES–OUTPUT
Desain sistem mengikuti format sederhana: **Input → Proses → Output**. Di dalam input/proses, bisa dibedakan mana kerjaan Can vs Dias pakai warna. Kalau perlu detail, bisa breakdown versi masing-masing di tahap selanjutnya, tapi untuk proposal cukup garis besar.
- **Sumber:** 16/6/26 (dukungan dari 13/3/26) — "Kalau design system itu intinya ada tiga, ada input, ada proses, ada output."

## 18. Metodologi Mix: Kuantitatif + Kualitatif
Output matriks/log = **kuantitatif** (angka). Penjelasan/analisa narasi = **kualitatif**. Narasi harus auto-generate menggunakan AI API (Gemini API). Jangan berhenti di angka saja, harus ada analisa.
- **Sumber:** 16/6/26 — "Matriknya itu yang kuantitatif. Yang kualitatif penjelasan terhadap matriknya itu. Narasi harus auto generate. Panggil API."

## 19. Hapus "Proses Human-Involved" dari Presentasi
Diagram yang menunjukkan proses "human-involved" tidak perlu dipresentasikan di PPT. Boleh masuk di proposal sebagai lampiran, tapi tidak perlu ditampilkan saat sidang.
- **Sumber:** 16/6/26 — "Enggak perlu. Kalau kamu mau masukin proposal bisa tapi enggak usah dipresentasikan."

## 20. Arsitektur Backend: Logging, Dashboard, Clustering
Backend flow: **Payload JSON → Validasi → SQLite → Admin Dashboard → Export Excel**. Clustering mengelompokkan user berdasarkan "percaya AI" vs "koreksi AI". Server menggunakan PRM (izin Pak Hafiz).
- **Sumber:** 16/6/26 — "Ini buat entry front end-nya terus masuk ke back end-nya itu pakai payload dari JSON-nya gitu. Terus nanti divalidasi... Store... Ini buat databasenya. Storage SQLite."
